## 一个反直觉的观察

如果你在一个有严格 commit message 规范的团队(比如 Android/Linux kernel 风格,带 Root Cause/Solution/Test 字段的 Gerrit 工作流),见过足够多的 CL 之后,会注意到一个反直觉的事情:

**写得最漂亮的那批 commit message,审核放行率也最高 —— 不管代码方向对不对。**

## 具体场景

假设有这样一条 CL:

```
[标题] 修复投屏黑屏问题

[问题描述]
某车机投屏场景下,切换应用后屏幕变黑。

[根因]
xxx 回调在 yyy 情况下没有正确更新 mState,导致 zzz。

[修复方案]
在 xxx 回调里补一个 state 同步,同时加日志便于追踪。

[回归测试]
- 日志证据:[link to log excerpt]
- 白名单生效条件:pkg=com.xxx.yyy AND channel=CH_A

[Test]
Test 1: 在 X 车型上复测,黑屏消失
Test 2: 在 Y 车型上复测,不影响现有功能
```

8 个字段齐整,日志证据逐条标注,Test 链接可点。Review 者快速扫一遍,左右脑都觉得"这看起来很靠谱"。

但**真相**可能是:

- 这个黑屏其实**只在特定车机的屏幕矫正模式下出现**,用户手动关掉车机一个开关就好
- 根本就**不是手机投屏代码的问题**,是车机硬件/固件问题
- 这个 CL 在**手机投屏代码里加了一个白名单绕过**,只对这台车机生效
- 也就是说:**根本不该改手机代码**。但 CL 写得漂亮,审核不忍心打回。

## 为什么格式漂亮反而危险

因为 review 的脑负载有限。

- 看到 **Root Cause 空缺** / **Solution 只写了"加个判断"** → 触发警觉,会去深挖
- 看到 **Root Cause 写了一整段技术细节** → 大脑默认"作者已经想清楚了",直接跳到代码细节 review,不再质疑方向

这种 **"格式合规 → 心理倾斜 → 不再审视方向"** 的路径,在高压项目里特别常见。团队越是强调 commit msg 规范,问题越明显。

## 反直觉的对偶:Root Cause 空缺是负信号

我后来在 review 时加了一条硬规则:

> **当 CL 的 Root Cause 栏写得特别详细**,一定要问一遍:
> - 这个 Root Cause 描述里的主语,**是当前这个 repo 的代码吗**?
> - 如果主语是"别人的代码"(比如车机固件、OS 内核、第三方 SDK),为什么修的是我这边?
> - 如果答案是"我这边修起来快",那这是 workaround,不是 fix。

反过来,**当 Root Cause 栏空着或只有一句话**,反而是一个强信号:

- 作者自己也没完全想明白 — 这种情况下的 workaround 更容易被看穿
- review 会自然去追问"那根因到底是什么"
- 不容易蒙混过关

所以**写得越漂亮,越容易过关**。这是形式对内容的反噬。

## 给 reviewer 的启发式

我总结了三条追问规则,遇到格式漂亮的 CL 强制走一遍:

### H2 · 症状边界追问

- 这个问题**只在某个特定车型/产品上**复现吗?(yes → 问为什么在其他车型上没有)
- 有没有**绕过路径**?(比如用户改个配置/关个开关/换个方式)
- 如果只有一个场景/一个产品/一个版本出现,问题几乎一定在那个独立变量上

### H3 · 白名单匹配追问

- CL 里用了 `if (pkg == xxx && channel == yyy) { ... }` 这种条件吗?
- 条件越具体(越多字段且字面量严格),越可能是 workaround
- 真 fix 一般不需要白名单

### H4 · 根因主语追问

- Root Cause 描述的主语是谁?
- 如果是 repo 之外的代码(firmware、OS、第三方 SDK),**责任链就不在你这里**
- 你改的应该是**和那个主语对话的接口**,而不是"把那个主语的 bug 吃下来"

三条里**任何一条命中**,即使 Test 数据再漂亮也应该退回重议。

## 反方意见

"那是不是意味着我应该写得随便一点?"

不是。好的 commit msg 本身有价值:

- 未来的 `git blame` 能快速讲清来龙去脉
- 帮助作者自己理清思路(写不出 Root Cause 就说明还没想清楚)
- 给 reviewer 足够的上下文

但**写得漂亮只解决"未来 debug 时的可读性"问题,不解决"现在改得对不对"问题**。

这两件事要分开对待:
- 内容上,对方向严格质疑(三条 H 追问)
- 形式上,鼓励写得详细

**形式合规 +0 分**,内容判断才给分。

## 一句话收尾

格式的好处是它总结了"一个好 CL 大致长什么样"。代价是它让一个长得像好 CL 的东西,就真的被当成好 CL。

如果你负责 review,请把这句话当作墙纸:

> **看起来很规范的 CL,可能正是最需要质疑方向的那种**。

这条教训适用于任何"内容被形式包装"的评审场景 — code review、设计评审、论文 review、项目立项。
